# Sourcing Real Per-District UHI Data for Bangkok

## Why

`data/gis/bangkok_uhi_data.csv` (generated by `data/fetch_district_data.py`'s
`create_mock_csv`) is currently a hand-coded placeholder, not measured data:
two hardcoded name lists (`high_uhi`, 9 districts; `low_uhi`, 7 districts),
sourced from an unspecified "World Bank report," get fixed labels
(`Severe`/`Low`); the other 34 districts all get an identical `Medium`
bucket. The `Death_Risk_Index` values (1.035 / 1.028 / 1.022) are constants
per tier, not independently sourced per district. This plan replaces that
with real, per-district measurements.

## Approach

Google Earth Engine (GEE), to avoid downloading/processing heavy raster
files locally. All zonal statistics computed server-side against
`data/gis/bangkok_districts.geojson` (the 50 real district polygons already
fetched from OSM).

## Update: the "World Bank report" was identified and its data extracted

`research/Modeling Spatio-Temporal Characteristics of Urban Heat in
Bangkok.pdf` (Acero, Sinh, Rubinyi; World Bank Policy Research Working
Paper 11158, June 2025) is the source the original mock script's comment
referred to. It uses dynamic WRF climate modeling (not satellite LST) to
compute real per-district Urban Heat Island Intensity (UHII, air
temperature vs. rural reference) for 50 Bangkok districts, across 3
seasons and multiple times of day, plus each district's real % share of
BMA population (Annex 1, Tables A1.1-A1.6).

Cross-checking: the original mock `high_uhi`/`low_uhi` lists turn out to
closely match this report's Table A1.1 (cool/dry season, night,
00:00-02:00) -- the report's own headline "most pronounced UHI" result.
So the mock data wasn't arbitrary, just a crude 3-tier simplification of
this real source that was never fully transcribed.

This also explains the Phase 1 LST finding: the report documents (Section
3.3) that Bangkok's compact high-rise core develops a daytime **"urban
cool island"** (tall buildings shade the ground from a nadir-viewing
satellite / the sun), while its UHI is dominantly a **nighttime air
temperature** phenomenon (up to 6.4°C above rural reference just before
sunrise). Landsat's ~10:30 AM overpass falls inside that documented
daytime cool-island window. Both datasets are real and correct -- they
measure genuinely different physical phenomena (surface vs. canopy/air
UHI), which is a well-documented effect, not a data quality problem.

`data/gis/bangkok_worldbank_uhii.csv` transcribes Table A1.1 (District,
UHII_Night_DecC, Population_Pct_BMA). `data/merge_uhi_data.py` was updated
to use this as the primary source for `UHI_Tier` (tertiles of the real
WRF nighttime UHII, not LST) since nighttime air temperature is what the
report ties to heat-mortality risk, and to add `Population_Pct_BMA`
directly -- covering Phase 3's population requirement from an
authoritative source, no WorldPop/GEE work needed. LST/LCZ are kept as
separate labeled columns (still real, still useful for this repo's
building-energy-simulation purpose) rather than discarded.

### Update 2: full seasonal/diurnal UHII table extracted

## Phase 0 — Trace the source claim

Find the actual World Bank report referenced in the original script's
comment. If it has real per-district figures, extract those directly
instead of rebuilding from satellite data. If it's only qualitative,
Phases 1-3 fully replace it.

## Phase 1 — Land Surface Temperature (LST) per district

The real physical UHI signal.

- **Source**: Landsat 8/9 Collection 2 Level 2, thermal band `ST_B10`
  (~30-100m resolution — fine enough for districts as small as ~1 km²,
  unlike MODIS's 1km grid).
- **Method**: cloud-masked (QA_PIXEL), multi-year dry-season composite
  (Mar-May, when Bangkok UHI is most pronounced) to smooth out single-scene
  noise. Convert `ST_B10` DN to Celsius:
  `Kelvin = ST_B10 * 0.00341802 + 149.0`, `Celsius = Kelvin - 273.15`.
- **Zonal stats**: mean/max LST per district polygon via
  `ee.Image.reduceRegions`.
- **Output**: `LST_Mean_C`, `LST_Max_C` per district, replacing the
  fabricated `UHI_Risk` label. A categorical `Severe/Medium/Low` field can
  still be derived afterward via quantiles of the real distribution, for
  the existing Grasshopper color-coding.
- **Status**: DONE. `data/fetch_uhi_lst.py` computes a 2019-2024 dry-season
  (Mar-May), cloud-masked Landsat 8/9 C2 L2 composite and writes
  `data/gis/bangkok_lst_data.csv` (50/50 districts, no missing data).
- **Important finding**: results contradict the mock data's assumptions.
  Don Mueang (mock-labeled "Low") is the hottest district (45.6°C mean,
  55.4°C max) — it contains Don Mueang International Airport, and bare
  tarmac/runway is a classic LST hotspot. Dense CBD districts the mock
  data called "Severe" (Pathum Wan, Bang Rak, Sathon, Samphanthawong,
  Khlong San) are actually among the coolest (41.8-42.9°C). This is a
  real, documented effect, not a bug: satellite-derived surface LST and
  street-level canopy UHI (what affects human comfort/mortality) diverge
  in high-rise cities, since tall buildings shade the ground from a
  nadir-viewing satellite even while trapping heat at pedestrian level.
  LST is nonetheless directly relevant to this repo's actual purpose
  (building energy simulation via EPW morphing), so kept as-is. Also note
  pixel counts vary hugely by district size (180 for tiny Samphanthawong
  vs 25,079 for sprawling Nong Chok) — small downtown districts' averages
  are statistically noisier.

## Phase 2 — Local Climate Zone (LCZ) per district

- Check first whether Bangkok already has a submitted map on the
  [WUDAPT LCZ Generator](https://lcz-generator.rub.de) / WUDAPT portal —
  if so, this phase is just downloading it and taking the zonal majority
  per district instead of building one from scratch.
- If not: derive a simplified proxy from building density + height (OSM
  building footprints + GHSL building-height layer) rather than running
  the full WUDAPT training pipeline.
- **Output**: `Dominant_LCZ` per district from actual built-form data,
  replacing the current hardcoded LCZ-1/3/6 tier assignment.
- **Status**: DONE. Bangkok is already covered by the published global LCZ
  map (Demuzere et al. 2022, `RUB/RUBCLIM/LCZ/global_lcz_map/latest` on
  Earth Engine) — no need to run the WUDAPT training pipeline ourselves.
  `data/fetch_uhi_lcz.py` computes zonal majority class + mean confidence
  per district and writes `data/gis/bangkok_lcz_data.csv` (50/50
  districts, no missing data).
- **Result**: much more varied and realistic than the mock 3-tier split —
  7 distinct LCZ classes actually present (LCZ 2 Compact midrise: 9
  districts, LCZ 3 Compact low-rise: 17, LCZ 5 Open midrise: 9, LCZ 6 Open
  low-rise: 7, LCZ 8 Large low-rise: 5, LCZ D Low plants: 2, LCZ C
  Bush/scrub: 1). Notably **no district classifies as LCZ 1 (Compact
  high-rise)** — even Bangkok's CBD (Pathum Wan, Bang Rak, Sathon,
  Samphanthawong) comes back as LCZ 2 (Compact midrise) or LCZ 3, which
  matches the LST finding above (these are actually cooler than assumed)
  and is more realistic than the mock data's LCZ-1 assumption for these
  districts. Classification confidence ranges 52-85%; lower-confidence
  districts (e.g. Ratchathewi 52.3%, Pathum Wan 56.4%) reflect genuinely
  mixed built form rather than a data quality problem.
- **Standalone check**: `data/gis/plot_lcz_map.py` renders
  `bangkok_lcz_data.csv` + `bangkok_districts.geojson` as a matplotlib
  choropleth (`--highlight "<name>"` outlines one district, `--out` saves a
  PNG) — a way to sanity-check any district's dominant class without going
  through Grasshopper. Confirmed via this that a district visually reading
  as "wrong" in a GH view (e.g. Sathon appearing as LCZ 6) was a
  GH-side matching/ordering issue, not bad source data — the CSV has always
  had Sathon as LCZ 3 (59.6% confidence).

## Phase 2b — Full-resolution LCZ grid (no per-district aggregation)

Per-district mode (Phase 2) collapses ~half a district's worth of pixels
into one class, which hides real intra-district variation (e.g. a compact
high-rise core surrounded by lower-density residential soi, like Sathon).
To instead map the **entire pixel grid** in Grasshopper — like a WRF/WUDAPT
LCZ input map, colored cell-by-cell rather than district-by-district — skip
the zonal reducer and pull the raw raster directly.

- **Data source**: same as Phase 2, `RUB/RUBCLIM/LCZ/global_lcz_map/latest`
  on Earth Engine, `LCZ_Filter` band (+ `LCZ_Probability`). No new dataset.
- **Fetch method**: `data/fetch_lcz_grid.py` uses `ee.data.computePixels()`
  — a single synchronous pixel-array request — instead of
  `reduceRegions()`/`Export.image.toDrive`. Region is small enough (Bangkok
  districts' bounding box) that even native 100m resolution (~341k cells)
  fits comfortably under computePixels' response-size limit; no Drive
  export or async task polling needed. Verified live: 200m default scale →
  330×259 = 85,470 cells in one call, output visually spot-checked against
  the WRF LCZ figure (matching river shape / dense core / green fringe
  pattern). Resolution is a tunable `--scale` flag in meters (100m ≈341k
  cells, 150m ≈152k, 200m ≈85k \[default\], 250m ≈55k).
- **Cost**: negligible — response is a few hundred KB to low single-digit
  MB even at 100m, far under any Earth Engine quota. No cost expected
  unless the `weather-data-morphing` GCP project is registered as
  commercial (unconfirmed — worth checking at
  `code.earthengine.google.com/register` if usage patterns change, e.g.
  many repeated fetches at very high resolution).
- **Coordinate system**: cell centers are projected into the same local
  planar XY (meters, equirectangular, origin = mean vertex of all district
  geometry) that `data/gis/gh_geojson_to_curves.py` already uses for the
  district curves, computed with the identical formula — so the grid lines
  up with existing GH geometry with no separate alignment/georeferencing
  step.
- **Output**: `data/gis/bangkok_lcz_grid.csv` (`Row,Col,X,Y,LCZ_Code,
  LCZ_Confidence_Pct`, one row per cell) + `data/gis/bangkok_lcz_grid_meta.json`
  (grid dimensions, scale, projection origin).
- **Grasshopper side**: `data/gis/gh_lcz_grid_mesh.py` reads both files and
  builds a **single Rhino `Mesh`** — one flat-colored quad per cell, vertices
  unwelded per cell so colors don't blend across cell boundaries (sharp
  pixel look, matching the WRF figure, instead of a smooth-shaded surface).
  Deliberately outputs one Mesh object rather than parallel point/color
  lists: `gh_geojson_to_curves.py` already documented that this Script
  component's output channel breaks on large lists (~1700+ items), and this
  grid has tens of thousands of cells.
- **Status**: fetch script (`fetch_lcz_grid.py`) run and verified against
  live Earth Engine data. `gh_lcz_grid_mesh.py` written and checked against
  the RhinoCommon API (`Mesh.Vertices.Add`/`Faces.AddFace`/
  `VertexColors.Add`/`Normals.ComputeNormals`) but **not yet executed
  inside actual Rhino/Grasshopper** — no Rhino available in the dev
  sandbox. Pending: user validation on the GH canvas (check the script's
  `Report` output first).

## Phase 3 — Heat-health / vulnerability risk

Real district-level mortality attribution is unlikely to be publicly
available for Thailand at this resolution.

- **First**: search Thailand MOPH and Bangkok Metropolitan Administration
  (BMA) publications, and academic papers, for district or sub-district
  heat mortality data.
- **Fallback**: build a transparent composite **Heat Vulnerability Index**
  instead of pretending to have mortality data: LST (Phase 1) × population
  density × % elderly population (≥65). Treat richer factors (income, AC
  ownership, healthcare access, informal housing) as stretch goals only
  if a clean source turns up — not required.
- **Output**: either real mortality data if found, or a documented,
  reproducible composite index replacing the fabricated
  `Death_Risk_Index` constants.
- **Partial status**: `Population_Pct_BMA` (real, per-district % share of
  BMA population) is now in `bangkok_uhi_data.csv`, sourced directly from
  the World Bank report's Annex 1 (see the update note above) — no
  WorldPop/GEE work needed for this part. Still open: % elderly
  population (age structure), and the actual mortality/vulnerability
  composite itself.
